Task: fix large-repo snapshot performance in `entire-sem`

Repo: `/Users/thomi/Projects/entire-sem`

Problem:
`entire-sem` cannot complete a semantic snapshot for `/Users/thomi/Projects/cli` in bounded time. The command stalls before writing any NDJSON output:

```sh
go run ./cmd/entire-sem snapshot --repo /Users/thomi/Projects/cli --format ndjson --no-network
```

Findings:
- Sampling shows nearly all CPU in `internal/sem/provider.go:buildRelations`.
- The hot path starts around `internal/sem/provider.go:437`.
- Current logic loops over every parsed symbol, then over every repo-wide short symbol name, then calls `containsIdentifier(block, name)`.
- `/Users/thomi/Projects/cli` has about 929 tracked files and 760 supported source files, so this repo-wide nested scan does not scale.
- Snapshot output is fully buffered until relation building finishes, so callers see no progress.

Goal:
Make full-repo snapshot generation complete reliably for `/Users/thomi/Projects/cli` without pruning, fake ignores, or reducing repository scope.

Suggested implementation:
- Replace the repo-wide `for name := range symbolsByShortName` scan with a bounded lookup.
- Extract identifiers from each symbol block once.
- For each identifier, look up `symbolsByShortName[identifier]`.
- Emit `CALLS` relations only for actual identifier hits.
- Add safeguards for ambiguous/high-fanout names if needed.
- Preserve existing relation types: `DEFINES`, `CONTAINS`, `IMPORTS`, `HANDLES_ROUTE`, and `HANDLES_TOOL`.

Tests:
- Add a regression test in `internal/sem` with many symbols and many files that would be slow under the old nested scan.
- The test should verify `CALLS` relations are still emitted for matching identifiers.
- The test should also verify unrelated symbol names are not scanned into relations.

Acceptance criteria:
- `go test ./...` passes.
- This completes in bounded time:

```sh
go run ./cmd/entire-sem snapshot --repo /Users/thomi/Projects/cli --format ndjson --no-network > /tmp/cli-snapshot.ndjson
```

- Record:
  - runtime
  - NDJSON line count
  - file count
  - symbol count
  - relation count
  - warning / partial failure count

Do not solve this by adding benchmark-only ignores, synthetic `.brainignore`, or skipping `/Users/thomi/Projects/cli` content.
