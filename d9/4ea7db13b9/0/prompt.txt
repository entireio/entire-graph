# Fix: SQL grammar can't parse PostgreSQL/Supabase migrations

  ## Context
  entire-sem uses `github.com/smacker/go-tree-sitter/sql` for `.sql` files
  (internal/sem/parser.go:79 — `".sql": {language: "SQL", grammar: sql.GetLanguage()}`).
  That is a generic SQL grammar with no PostgreSQL support. On real Supabase/Postgres
  migration repos it produces tree-sitter ERROR nodes on almost every file, so the
  file is reported as an E_PARSE_ERROR partial failure and its symbols/relations are
  incomplete.

  ## Evidence (reproduced)
  On the `podcasts` repo (~/Projects/podcasts), 125 of 133 .sql files fail
  (`entire-sem snapshot --repo . --format ndjson --no-network` →
  header.stats.partial_failures = 125, completeness_level "unsafe"). Parsing
  `supabase/migrations/202605090001_phase2_contract.sql` with sql.GetLanguage()
  returns HasError()=true with the first ERROR node at `do $$` (line 4). Unsupported
  constructs include:
  - dollar-quoted blocks: `do $$ ... $$;`
  - `create or replace function ... language plpgsql as $$ ... $$;` (PL/pgSQL bodies)
  - `create extension ...`
  - generated columns: `... generated always as (...) stored`
  - pgvector types: `vector(1536)`, plus `timestamptz`, inline `check (... in (...))`,
    `on conflict ... do update`
  The only 8 passing files contain none of these (plain update/create index/insert).

  ## Goal
  Parse PostgreSQL (esp. Supabase migrations) without ERROR nodes so symbols/relations
  are extracted and E_PARSE_ERROR rates drop to near zero on Postgres repos.

  ## Suggested approach
  1. Replace/augment the SQL grammar with a PostgreSQL-aware tree-sitter grammar
     (e.g. DerekStride/tree-sitter-sql, which handles dollar-quoting, PL/pgSQL, DDL).
     Confirm a Go binding is available or add one; weigh against the existing
     smacker dependency set.
  2. If a single grammar can't cover all dialects, consider dialect selection
     (e.g. Postgres grammar for files under supabase/migrations/ or by content
     sniffing for `$$`/`language plpgsql`).
  3. Make sure walkEntities/entitySymbols know the new grammar's node types
     (function/table/view/index/policy/trigger) so symbols are actually emitted,
     not just error-free parses.

  ## Acceptance criteria
  - `entire-sem snapshot` on ~/Projects/podcasts reports partial_failures ≈ 0
    (was 125) and completeness_level "ok".
  - Symbols are emitted for Postgres functions/tables/policies (spot-check
    202605090001_phase2_contract.sql yields table + function symbols).
  - Existing non-SQL parsing and tests are unaffected; add a Postgres SQL fixture
    test under internal/sem.
  - No new network dependency at runtime (provider must stay no-egress;
    `entire-sem doctor` keeps reporting no_egress=true).

  ## Out of scope
  The path-loss bug on the consumer side is being fixed separately in entire-brain
  (JSON key mismatch: provider emits `file_path`/`effect_on_semantic_completeness`,
  entire-brain expected `path`/`effect`). Keep emitting the current provider keys.
