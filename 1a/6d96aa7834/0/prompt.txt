## Finding 1 — CRITICAL: `git grep -I` silently drops binary-flagged files the old scan counted; the "strict superset" comment is false

**Location:** `/Users/alex/workspace/entire-graph/internal/sem/dependents.go:99-121` (new prefilter) + `/Users/alex/workspace/entire-graph/internal/gitutil/git.go:139` (the `-I` flag in `GrepTreePaths`)

The doc comment on `referenceCandidateFiles` claims the grep pass "can only add extra candidate files, never drop a real dependent." That claim is only true of the `-i -F` substring semantics. It is defeated by the `-I` flag, which excludes any file git classifies as binary. Git classifies a blob binary if (a) it contains a NUL byte in roughly the first 8KB, or (b) `.gitattributes` marks it `binary` or `-diff`. The old path (`ListFiles` + `ShowFile` + tree-sitter parse) had no such exclusion — it parsed these files and counted dependents in them.

I confirmed this empirically in a scratch repo (`git grep -z -I -i -F -l -e helper HEAD`):
- a `.js` file with one embedded NUL byte in a string literal: **dropped** by `-I`, listed without it
- a clean text `.js` file with `marked.js binary` in .gitattributes: **dropped**
- `a.js -diff`: **dropped**; `-text`: not dropped
- worse: a **worktree-only, uncommitted** `.gitattributes` changes the result of grepping a committed historical tree — dependent counts for the same commit now differ between checkouts

**Hidden failure scenario (realistic, not contrived):** `*.pb.go -diff` and `*.min.js -diff` are widespread `.gitattributes` conventions for generated code. **This repo today:** not affected — I diffed `git grep -a -l` vs `git grep -I -l` over all 269 files at HEAD: zero binary-flagged files, no `.gitattributes` exists, and none of the vendored `parser.c` files trip the heuristic. **Recommendation:** drop `-I` for the dependents prefilter.
