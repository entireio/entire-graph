## Summary of what the commit does

Broadens CALLS-edge resolution across many languages. Three mechanism classes:
1. **New call-name scanners** (`call_scanners.go`, plus `rust.go`/`php.go`/etc. helpers) that produce *candidate* call names for Clojure, SQL, Objective-C, JS-dotted, F#, Julia, Lua-dotted, Zig-dotted, Rust turbofish/module-path.
2. **New resolution tiers** in `provider.go`/`receiverCallRelations` (JS export fallback, SQL duplicate fan-out, TS interface-impl, Perl/Zig/Rust unique-name, PHP factory chains, TS DI receivers).
3. **Import-graph broadening** (Rust/Kotlin imported names, JS `export *` re-export closure, dotted-module file matching, `.js`тЖТ`.ts` candidate mapping, un-ignoring via explicit ignore files applied to the HEAD path).

Key structural fact that bounds most scanner risk: the new scanners only add **candidate names**; a false candidate becomes a false CALLS edge only if a real workspace symbol shares that name *and* passes the downstream resolution gates. So scanner over-matching is generally low-risk in isolation; the higher-risk items are the resolution tiers and the broadest scanners (bare-argument, dotted).

## Concrete issues (file:line, why, severity)

**1. `jsCallableArgumentIdentifiers` тАФ MEDIUM (highest scanner risk)**
`internal/sem/call_scanners.go:91` (wired in at `provider.go:2521`/`2529` and `2872`+). Every bare identifier passed as a call argument (`save(user)`, `handle(next)`, `run(error)`) is treated as a callable. This cannot distinguish a function reference (`map(callback)`) from a plain value argument (`save(user)`). At scale, common identifiers (`next`, `error`, `data`, `handler`, `value`) that are simultaneously variable names and function names elsewhere in the workspace will mint false CALLS edges. Only JS/TS-gated and bounded by name-collision + resolution, but this is the one scanner that systematically feeds non-call constructs into the resolver. Precision-negative for large JS/TS repos.
