# entire-graph: Exploration Report

## 1. What the tool is and its purpose

**entire-graph** (product name "Entire Graph"; module `github.com/entireio/entire-graph`) is a **deterministic, local-first code graph for coding agents**, and the semantic layer behind Entire checkpoints. It parses a repository with tree-sitter and answers structural questions (where is X defined, who calls it, what changed at the entity level, what will break) from a persistent knowledge graph of functions, classes, call chains, routes, and cross-service links.

Key properties (README.md:11-24, 51-62):
- **No LLM, no embeddings, no vectors, no network.** Pure tree-sitter static analysis. Same commit always yields the same graph.
- **100% local / zero egress.** No API keys, no telemetry, no grammar downloads at runtime. `entire graph doctor --json` reports `no_egress=true`.
- **Git-native.** Every snapshot keyed by `(repo, commit, tree)`; reads committed `HEAD` by default, `--worktree` for live edits.
- **36 semantic languages + 149 inventory-only filetypes** (185 recognized names), all from vendored grammars compiled into one binary.
- **30 relation types** (Schema 1.1) with stable `compound-v1` symbol IDs that survive ordinary edits.

Primary docs: `README.md` (very thorough, 470 lines), `AGENTS.md` (agent-facing instructions), and `docs/` (language-support.md, benchmarks.md, operations.md, semantic_provider_requirements.md, adr/0001-ga-schema-contract.md).

## 2. How it's invoked

It ships as an **Entire CLI plugin**. The manifest `entire-plugin.yml` registers the subcommand name `graph`, so users type **`entire graph &lt;command&gt;`**. The binary itself is `entire-graph` (entry point `cmd/entire-graph/`; all logic in `internal/cli/`). Note: uses hand-rolled dispatch/flag parsing, no cobra/CLI framework (AGENTS.md:60-63).

## 3. Main workflows / use-cases

1. **Semantic diff (judge a change):** `entire graph commit HEAD --json` or `diff --base A --head B`. Extracts named entities from before/after git trees, classifies each as added/removed/renamed/signature-changed/body-changed, attaches heuristic dependent counts. Lets an agent decide keep/revert/continue without reading the full textual diff (README.md:223-233).
2. **Search (find where to work):** `entire graph search --repo . --query "..." --format agent`. Ranked, diverse source regions fusing body matching, identifier splitting, symbol/signature/path signals, graph-neighbor expansion. Working tree by default; `--head` for committed semantics. Output budgeted to 16 KiB (README.md:146-193).
3. **Neighbors (structural follow-up):** `entire graph neighbors --symbol X --file f --depth 2 --format agent` — resolved incoming/outgoing relations + two-hop caller→focus→callee paths (README.md:195-219).
4. **Build a code graph (snapshot streaming):** `entire graph snapshot --repo . --format ndjson` — header + file + external-endpoint + symbol + relation records, streamed. For ingestion into agent memory or Entire Brain (README.md:287-291).
5. **Pre-indexing:** `entire graph index --head --profile full --cache-dir ...` before latency-sensitive agent work (README.md:148-165).
6. **Feature-detect:** `capabilities --json` / `doctor --json`.

## 4. Config, env vars, setup

**Environment variables** (`internal/cli/env.go:6-8`):
- `ENTIRE_CLI_VERSION` — set by parent Entire CLI.
- `ENTIRE_REPO_ROOT` — repo root when running outside Entire (or use `--repo`).
- `ENTIRE_PLUGIN_DATA_DIR` — enables the tree-keyed compressed index cache for committed-tree index/search/neighbors.

Inside Entire sessions, `ENTIRE_REPO_ROOT` and `ENTIRE_PLUGIN_DATA_DIR` are set automatically. Outside Entire, pass `--repo` (or set `ENTIRE_REPO_ROOT=/path ./entire-graph diff ...`) (README.md:138-142, 294).

From source: `mise run build` then `entire plugin install ./entire-graph --force`, or `scripts/install-local.sh`. **Requirements:** Entire CLI, Git, Go 1.24 toolchain with **CGO enabled** (tree-sitter native bindings). Build tasks are in `mise.toml` (`mise run build|test|check`). No config files beyond gitignore-style `--ignore-file`/`--include-file` refinements; snapshots honor root `.gitignore` by default.

Note the rename: the plugin was `entire-sem` before v0.2.0; `@latest` needs v0.2.0+.

## 5. Output formats

- **Human text** (default for diff commands) — e.g. `~ function validate_token signature changed (14 dependents)` (README.md:129-136).
- **`--json`** — structured single-object output for diff/doctor/capabilities/version.
- **`--format ndjson`** — streamed graph records for `snapshot`/`symbols`/`edges`: one header record then file, external-endpoint, symbol, relation records (README.md:371-383).
- **`--format agent`** (search/neighbors) — compact, telemetry-prefixed, byte-bounded output sized for a model context window (default 16 KiB); never exceeds requested bytes.
- **`--format text`** — human-readable for search/neighbors.

Schema is a **frozen GA contract, schema 1.x** (additive-only minors; breaking changes need a major bump) — see `docs/adr/0001-ga-schema-contract.md` (AGENTS.md:71-82). Headers carry schema version, provider version, repo key, HEAD commit/tree, parsed languages, capability labels, warnings, partial failures, completeness stats. Unsupported/unparseable files surface as machine-readable partial failures, never silent drops.

## 7. Relationship to the "entire" CLI / Entire platform

- entire-graph is an **Entire CLI plugin**: it registers subcommand `graph` via `entire-plugin.yml`, and is invoked as `entire graph ...`. Entire discovers the `entire-graph` binary in its managed plugin directory or on `$PATH` (README.md:83).
- It "requires no changes to the main Entire CLI" (README.md:463).
- It **doubles as a local-only semantic provider** feeding **Entire Brain** (`entire-brain`) — a separate downstream consumer that owns persistence, indexing, and query. entire-graph produces the graph; Brain stores/queries it (AGENTS.md:14-16, README.md:13).
- The same graph "powers entity-level checkpoint context, so a checkpoint can be judged (keep, revert, continue) without reading the full diff" (README.md:62). The `checkpoint &lt;id&gt;` command resolves an `Entire-Checkpoint: &lt;id&gt;` git trailer.
- It can also run fully standalone outside Entire via `ENTIRE_REPO_ROOT` / `--repo`.

# entire-graph performance-fix investigation

Branch `perf/graph-cache-reuse`. All line refs are current.

## FIX 1 — make the diff path reuse cached parse results

### Q1. Exact semantics of `dependents_count` (internal/sem/dependents.go)

Entry point `addDependentCounts` (dependents.go:15). Flow:

- **`names`** = `changedReferenceNames(result)` (dependents.go:36-47): the set of *short* names of every `EntityChange` in the result. Each name comes from `referenceName(change)` (dependents.go:123-134): for `renamed` it uses `NewName` (else `OldName`); otherwise `Name` — then passed through `shortEntityName` (dependents.go:136-141), which strips everything up to and including the last `.` (so `"User.validate"` → `"validate"`).
- **`buildReferenceIndex`** (dependents.go:49-89): lists **every** file in the head tree via `gitutil.ListFiles` (dependents.go:50), and for each `Supported` path, does `gitutil.ShowFile` + `parser.Parse`, splits content into lines, and for each parsed entity computes `entityBlock` (dependents.go:91-104 = `lines[StartLine-1:EndLine]`, the entity's own source text).
- **What counts as a dependent** (dependents.go:75-84): for each entity block and each `name`, it records a dependent — keyed `path+"#"+Kind+":"+Name` (a *set*, so each entity counts at most once) — when `containsIdentifier(block, name)` is true. `containsIdentifier` (dependents.go:106-113) tokenizes the block with `identifierBoundary = [A-Za-z0-9_$]+` (dependents.go:11) and matches a **whole token, case-sensitively** equal to `name`.
- **Exclusion / self-match** (dependents.go:78): an entity is skipped when `shortEntityName(entity.Name) == name`. This is a **by-name** exclusion, not by-identity — so the changed symbol's own definition site is excluded, *and so is every other entity that happens to share that short name*. There is no scope/language awareness; the match is purely lexical.
- `DependentsCount` is then `len(index[name])` per change (dependents.go:30).

**Output/tests depending on it:** field `EntityChange.DependentsCount` (model.go:31), JSON key `dependents_count`, emitted **always** (no `omitempty`). Text rendering `dependentSuffix` → `" (N dependents)"` (text.go:95-98, used at text.go:65/162). Tests: `TestAnalyzeGitRangeDependentCounts` (analyze_test.go:116-150, asserts `validate_token` → 1), plus references in text_test.go and internal/cli/root_test.go. It also participates as the final tiebreaker in `sortChanges` (analyze.go:469).

### Q2. What the snapshot stores — and whether content is present

`ProviderSnapshot` (provider.go:279-285): `Header`, `Files []FileRecord`, `Externals`, `Symbols []SymbolRecord`, `Relations []RelationRecord`.

- **`SymbolRecord`** (provider.go:218-236) carries: `Kind`, `Name`, `QualifiedName`, `FilePath`, `StartLine`, `EndLine`, `Signature`, `BodyHash`, `Language`, `ContainerID`, `Local`. So symbols **do** carry name/kind/file path/start+end lines/signature — but only a **`BodyHash`**, never body text.
- **`FileRecord`** (provider.go:193-201): `ID`, `Path`, `Blob` (blob SHA), `Language`, `Bytes`, `Lines`. No content.
- `newCachedSearchSnapshot` (search_cache.go:222-247) stores the `ProviderSnapshot` plus two internal-only maps (`FileLines`, `LocalSymbolIDs`) that are otherwise stripped from the wire format.

**Critical answer: neither file content nor entity body text is stored anywhere** — only metadata, boundaries (lines), signatures, hashes, and the blob SHA.

### Q3. Can `buildReferenceIndex` be reproduced from the snapshot alone?

**No.** `containsIdentifier` needs the actual block *text* (or at least its identifier-token set); the snapshot has only `BodyHash`. Evaluated options:

- **(a) cache the `referenceIndex` keyed by tree** — weak: the index is a function of `names`, which is the *diff-specific* changed set and varies per invocation. You'd have to cache something query-independent instead.
- **(b) cache parsed entities + per-entity identifier-token sets, keyed by blob SHA** — cleanest incremental win, exactly semantics-preserving. `identifiersIn` already exists (dependents.go:115-121) and is the precise token set `containsIdentifier` tests membership against. Store per blob: `{Kind, Name, StartLine, EndLine, identifierSet}`; only changed blobs re-parse. `FileRecord.Blob` (provider.go:197) gives the SHA. Membership test `name ∈ identifierSet` is *identical* to `containsIdentifier`. Larger diff, biggest long-term payoff.
- **(c) git-grep prefilter + parse only matching files** — **smallest diff, semantics-preserving, recommended first step.** `buildReferenceIndex` only needs files whose text contains one of the `names` tokens. Restrict `gitutil.ListFiles` to `gitutil.GrepTreePaths(ctx, repo, head, names)` (git.go:129) — the exact mechanism search's own preselection uses (`searchGitGrepPatterns`/`GrepTreePaths`, search.go:824-826). Note `GrepTreePaths` uses `-F -i -l` (fixed-string, **case-insensitive**, substring) — a **superset** of the case-sensitive whole-token match, so the candidate set can only be larger than needed; the subsequent `containsIdentifier` still applies exact semantics, so **counts are unchanged**. On a typical diff this collapses "parse every supported file in the tree" to "parse the handful containing a changed name."

Recommendation: ship (c) now (few-line change, contract-safe), layer (b) later for the incremental case.

### Q4. Where does the diff path get a `cacheDir`?

Today it doesn't. `runCommit` (root.go:359), `runDiff` (root.go:405), `runCheckpoint` (root.go:382) call `analyzeAndPrint`→`sem.AnalyzeGitRange` (root.go:450-451) / `sem.AnalyzeCheckpoint` (root.go:394) with **no cache parameter**, and `parseCommonFlags` only yields `Repo`/`JSON` (no `--cache-dir`/`--no-cache`).

The plumbing exists, though: `Options.Env` (root.go:17-22) → `EntireEnv.PluginDataDir` (env.go:16, from `ENTIRE_PLUGIN_DATA_DIR`). search/neighbors/index already do `cacheDir := flags.CacheDir; if "" { cacheDir = opts.Env.PluginDataDir }` (search.go:54-56, neighbors.go:118-120, index.go:58-60). To thread it: resolve `cacheDir` the same way in `runCommit/runDiff/runCheckpoint`, pass it through `analyzeAndPrint` → `AnalyzeGitRange`/`AnalyzeCheckpoint` → `addDependentCounts` → `buildReferenceIndex`. (Optionally add matching `--cache-dir`/`--no-cache` flags for parity.)

### Q5. Concurrency

`buildReferenceIndex` is **single-threaded** — a plain sequential loop over files (dependents.go:61-86). There is **no worker-pool parsing to borrow**: `grep` for `errgroup|WaitGroup|go func|NumCPU|semaphore` across `internal/sem/*.go` (non-test) matches **only search.go**, and **not** provider.go's build path — so the provider snapshot build is also serial. Parallelizing would be net-new work; the far bigger win is eliminating the reparse (cache reuse) or shrinking the file set (option c), not threading the existing loop.

## FIX 2 — key/validate the snapshot on TREE only (not commit+tree)

### Q6. Every place the commit SHA participates, and where a stale commit would leak

**Key / validation:**
- `searchSnapshotKey` writes the commit into the hash: `writePart(commit)` (search_cache.go:478). Removing this makes the key tree-only.
- `validCachedSearchSnapshot` checks commit twice: `cache.Commit == commit` (search_cache.go:516) and `cache.Snapshot.Header.Commit == commit` (search_cache.go:520). Both must be relaxed to tree-only.
- `cachedSearchSnapshot.Commit` struct field (search_cache.go:23) and `SnapshotHeader.Commit` (provider.go:129) both persist the build-time commit.

**HEAD-changed / provenance guards that would falsely trip if a commit-A snapshot is loaded at commit B (same tree):**
- search no-hit path: search.go:288-290 (`cachedSnapshot.Header.Commit != selection.commit` → returns "repository HEAD changed during search").
- search main path: search.go:360-362 (same guard).
- `exactFullPreindex` fast-path gate: search.go:820-823 (`preindexedSnapshot.Header.Commit == source.commit`) — a stale commit disables the git-tree-grep fast path.
- `selectiveSearchSnapshotFromFull` provenance check: search_cache.go:280 (`sc.commit != full.Header.Commit`).
- `PreindexProviderSnapshot` / `loadOrBuildSearchSnapshot` provenance asserts: search_cache.go:151, 198.

All of these compare a stored `Header.Commit` against the *current* commit and must be switched to compare **Tree** only.

**Where the stale commit would leak into user-visible output** (if not patched):
- Search: `SearchResponse.Commit` (search.go:313 and 577; no-hit path assigns from `cachedSnapshot.Header.Commit` at search.go:300) → JSON `"commit"` in CLI output (search.go:98). Applies equally to `agent`/`text`/`ndjson` formats that echo it.
- Neighbors: `neighborResponse.Commit = snapshot.Header.Commit` (neighbors.go:322) → JSON/agent output.

**Required patch on load:** extend `restoreCachedSearchInternals` (search_cache.go:249-260) — or the load sites in `readSearchSnapshot` (search_cache.go:546) — to **re-stamp** the current requesting commit onto `cache.Commit` and `cache.Snapshot.Header.Commit` after a tree-only hit, so a tree-keyed entry never emits the stale build-time commit. (Tree stays as stored; commit is re-derived from the caller's `resolveCommittedHEAD` — provider.go:1283.) This keeps the header field honest, which the GA contract requires (see below).

### Q7. Tests pinning current key composition / validation

- **`TestPreindexProviderSnapshotDoesNotReuseSameTreeAcrossCommits`** (search_cache_test.go:507-549) — **directly contradicts FIX 2.** It commits `--allow-empty` (same tree, new commit) and asserts the second preindex is **not** a cache hit (search_cache_test.go:538-540) and that `Header.Commit` differs. FIX 2's entire purpose is to make this a cache hit; this test must be rewritten to assert the *opposite* (same-tree → hit, with commit re-stamped).
- `TestSearchSnapshotCacheKeyPreservesIgnoreFileOrder` (search_cache_test.go:551) and `TestPreindexProviderSnapshotServesSelectiveSearch` (search_cache_test.go:162, key built at :204) both call `searchSnapshotKey(..., commit, tree, ...)` — a signature change (dropping `commit`) ripples into these.
- Numerous tests read `Header.Commit`/`Header.Tree` for provenance assertions (e.g. search_cache_test.go:204, 541-547, 581-585).

## GA schema contract (docs/adr/0001-ga-schema-contract.md) — what we may NOT change

Schema is frozen at **1.x, additive-only** (ADR §Decision): a minor may add optional fields/record kinds but may **never remove a field, change a field's meaning, or make an optional field required**; breaking changes need a `2.0` bump.

Applied here:
- **`dependents_count`** is an emitted field with a fixed meaning. FIX 1 must be **exactly semantics-preserving** — any caching or git-grep prefilter must yield *identical* counts. This is why option (c)'s over-approximating candidate set is safe (final `containsIdentifier` is unchanged) and why (b) must faithfully reproduce the `identifiersIn` token set. We may **not** redefine what it counts, drop it, or make it conditional.
- **Header fields** (`commit`, `tree`, `provider`, `profile`, …, provider.go:123-148) are part of the contract — cannot be removed or renamed. For FIX 2 this means `commit` must remain present **and meaningful**: emitting a stale build-time commit for a different HEAD arguably violates the field's meaning, so the on-load re-stamp (Q6) is contract-mandated, not optional.
